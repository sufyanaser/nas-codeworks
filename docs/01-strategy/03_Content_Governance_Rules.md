# NAS CodeWorks — Website Content Governance Rules

## Purpose

These rules prevent future copy, UI, CTAs, or marketing material from pushing NAS CodeWorks into the wrong category.

NAS CodeWorks must not drift toward:

- Software Company
- SaaS Company
- Digital Agency
- Freelancer Brand
- Warehouse Software Company

## Mandatory message order

Problem
→ Impact
→ Understanding
→ Suitable Path
→ Outcome
→ Trust
→ Problem Description CTA

Do not reverse this order.

## Language rules

Use:

- simple professional Arabic
- light Iraqi-white tone
- business-owner language
- short clear sentences
- practical examples
- daily work vocabulary

Avoid:

- heavy formal Arabic
- bureaucratic language
- technical jargon
- startup hype
- generic digital transformation phrases
- software-company language

## Problem-first rules

Every page must start from a problem or operational tension.

Never start from:

- service list
- technology
- app screenshots
- dashboards
- pricing
- features

## Service communication rules

Services are paths, not products.

Each service must be presented through:

Problem
→ Daily pain
→ Service path
→ Concrete outcome

## Technology communication rules

Technology may be mentioned only if it helps explain implementation or delivery after trust is already built.

Technology must never be the hero message.

## Case study rules

Case studies are not portfolio pieces.

They must follow:

Problem
→ Previous state
→ Impact
→ Solution path
→ Outcome

## Contact page rules

The contact page is a Problem Discovery Entry Point.

It must not become:

- Request a quote
- Book consultation
- Request software
- Project inquiry

## Pricing communication rules

Publish only approved starting prices.

Do not publish a number for Automation / Reporting before final commercial decision.

## Final content test

Any new content must answer:

Does this make the visitor feel that NAS CodeWorks understands their operational problem before trying to sell a tool?

If yes, it is aligned.
If no, reject it.
