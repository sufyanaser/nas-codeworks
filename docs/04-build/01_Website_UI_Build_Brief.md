# 2. Core UI Concept

## التحول البصري المعتمد

Fragmented Signals
→ Operational Center
→ Relationship Mapping
→ Structured Operations
→ Operational Clarity

## قاعدة حاكمة: المركز أهم من العناصر

هذه القاعدة تحكم كل قرار بصري ورسائلي في الموقع.

العناصر المتفرقة — Excel، WhatsApp، الملفات، التقارير، الموظفون — ليست الفكرة الرئيسية.
هذه إشارات تشتت ثانوية، وليست رموز الهوية.

الفكرة الرئيسية: غياب مركز يرى العلاقة بين هذه الأجزاء.

لذلك يجب أن يكون المركز التشغيلي هو المرساة البصرية الأقوى في كل رسم أو Hero أو diagram.
العناصر المتفرقة تظهر كـ fragments أو nodes ثانوية تتجه نحو المركز — لا كرموز مستقلة.

إذا تذكّر الزائر "هذه الجهة تحل مشكلة Excel"، فالتصميم فشل.
إذا تذكّر "هذه الجهة تساعدني أرى الصورة الكاملة لعملي"، فالتصميم نجح.

## معنى ذلك في الموقع

الموقع يبدأ من غياب مركز الرؤية التشغيلية:

* المعلومات موجودة، لكنها لا تجتمع في مكان واحد.
* الموظفون يعرفون أجزاء.
* الملفات والرسائل والتقارير تحمل أجزاء أخرى.
* لكن لا أحد يرى العلاقة بينها كلها في وقت واحد.

المشكلة ليست نقص المعلومات. المشكلة غياب المركز.

بعدها يظهر NAS CodeWorks كجهة تساعد على:

* كشف العلاقات بين الأجزاء المتفرقة.
* بناء مركز رؤية تشغيلي يجعل الصورة الكاملة مفهومة.
* تحديد مكان التعطيل الحقيقي.
* بناء أداة مناسبة عند الحاجة.
* الوصول إلى وضوح تشغيلي أفضل.

# 6. Global UI Principles

# 6.1 Problem First

كل صفحة يجب أن تبدأ من مشكلة أو توتر تشغيلي.

لا تبدأ الصفحة من:

* خدمة.
* تقنية.
* صورة واجهة.
* Dashboard.
* سعر.
* قائمة Features.

---

# 6.2 Operational Center Before Dashboard

لا يبدأ الموقع من شاشة برنامج أو Dashboard.

يبدأ من الفكرة:

المعلومات موجودة، لكنها متفرقة، ولا يوجد مركز واضح يجمع الصورة.

الـ Dashboard أو الواجهة، إذا ظهرت لاحقاً، تكون نتيجة لا بداية.

---

# 6.3 Clarity Over Decoration

كل عنصر بصري يجب أن يساعد على فهم:

* المشكلة.
* التشتت.
* المركز.
* المسار.
* النتيجة.

أي عنصر للتجميل فقط يجب حذفه.

---

# 6.4 Arabic-First

الموقع عربي أولاً.

يجب أن يكون:

* RTL كامل.
* قراءة مريحة.
* محاذاة عربية طبيعية.
* أرقام ومعلومات واضحة.
* نماذج سهلة لصاحب شركة غير تقني.

---

# 6.5 Calm Structure

الواجهة يجب أن تكون:

* هادئة.
* واسعة.
* منظمة.
* غير مزدحمة.
* ذات تسلسل واضح.
* بدون زحمة بطاقات أو حركات أو ألوان.

# 7. Visual System Translation

# 7.1 Operational Clarity

يُترجم عبر:

* تباين واضح.
* مساحات كافية.
* أقسام مرتبة.
* جمل قصيرة.
* بطاقات معلومات محدودة.
* خريطة رحلة واضحة.
* انتقال بصري من التشتت إلى التنظيم.

---

# 7.2 Operational Center

يُترجم عبر:

* عنصر مركزي بصري — هو المرساة الأقوى في الرسم، الأوضح والأثقل بصرياً.
* fragments ثانوية (ملف، تقرير، رسالة، موظف) تظهر كـ nodes خفيفة أو labels صغيرة.
* مسارات علاقة واضحة تصل الـ fragments بالمركز — لا مجرد تجميع.
* إظهار أن المركز يرى العلاقة بين الأجزاء، لا أنه يجمعها فقط.
* الانتقال من Fragmented Signals إلى Operational Understanding يجب أن يُقرأ بصرياً.

**تسلسل الأولوية البصرية:**
المركز أولاً ← مسارات العلاقة ثانياً ← الـ fragments ثالثاً.

المركز لا يجب أن يكون دائرة إذاعية.

إذا استُخدمت الدائرة، يجب أن تنبثق من العلاقات بين الأجزاء — لا أن تكون وعاءً يحتويها، ولا رمز بث أو موجة صوت.

---

# 15. Home Page UI Brief

# هدف الصفحة

جعل الزائر يشعر:

“هذه الجهة تفهم المشكلة التي أعيشها.”

# Hero

## يجب أن يحتوي

* عنوان يبدأ من مشكلة تشغيلية.
* فقرة توضّح التشتت.
* Visual يدعم فكرة المعلومات المتفرقة ومركز الرؤية.
* Primary CTA: ابدأ بوصف المشكلة.
* Secondary CTA: تعرّف على طريقة العمل.

## Hero Visual

**القاعدة الحاكمة: المركز هو البطل، ليس العناصر.**

الاتجاه الصحيح:

* المركز التشغيلي هو العنصر البصري الأقوى والأوضح في الرسم.
* الـ fragments (ملف، تقرير، رسالة، موظف، متابعة) تظهر كـ nodes خفيفة أو labels ثانوية.
* مسارات علاقة واضحة تربط الـ fragments بالمركز — تُظهر أن المركز يرى العلاقة بينها.
* لا يكفي أن تتجه العناصر نحو المركز — يجب أن يبدو المركز وكأنه يفهم ما يتلقاه.
* الـ fragments لا تحمل أيقونات برامج حقيقية — تكون أشكالاً هندسية خفيفة مع label نصي فقط.

## يمنع

* أيقونات Excel وWhatsApp الرسمية أو logos برامج فعلية.
* Dashboard ضخم.
* صورة برنامج.
* Mockup لابتوب.

* صورة فريق.

# 20. Components Required

## Global Components

* Header.
* Footer.
* Primary CTA Button.
* Secondary CTA Button.
* Section Heading.
* Problem Card.
* Outcome Card.
* Service Path Card.
* Process Step.
* Case Story Teaser.
* Contact Form Field.
* Status / Highlight Badge.
* Simple Diagram Block.

## Visual Components

* Operational Center Diagram — المركز هو العنصر الأقوى بصرياً، الـ fragments ثانوية.
* Relationship Paths — خطوط تربط الـ fragments بالمركز، تُظهر العلاقة لا مجرد القرب.
* Scattered-to-Structured Motif — انتقال من Fragmented Signals إلى Operational Center إلى Clarity.
* Missing Center Block — يُظهر الأجزاء المتفرقة بدون مركز (الوضع الحالي).
* Workflow Path.
* Before / After Block.
* Information Layer Stack.
* Simple Status Card.

# 21. Content Integration Rules

## يجب استخدام النسخ المعتمدة

* Home Page V3.
* Services Page V2.
* How We Work Page V2.
* Contact Page V2.
* Case Studies Page V2.

## لا يجوز

* إعادة كتابة النصوص أثناء التصميم.
* اختصار الرسائل بطريقة تكسر التموضع.
* تغيير CTA.
* تحويل المحتوى إلى Features.
* إضافة خدمات.
* نشر سعر Automation.

## يسمح فقط

* تقطيع النصوص إلى كتل UI.
* توزيع الفقرات بصرياً.
* إضافة Microcopy داعم إذا كان يخدم الفهم.
* تخفيف التكرار البصري دون تغيير المعنى.

# 22. Technical Build Direction

## القرار المقترح للنسخة الأولى

موقع ثابت أو شبه ثابت، سريع وخفيف، Multi-page، Arabic-first.

## الأولوية

* سرعة تحميل.
* SEO أساسي.
* RTL صحيح.
* Accessibility.
* Form قابل للإرسال.
* Mobile-first.
* سهولة التعديل لاحقاً.

## لا يحتاج الإصدار الأول إلى

* CMS معقد.
* Dashboard إداري.
* تسجيل دخول.
* Blog.
* SaaS app shell.
* Animations ثقيلة.
* مكتبة مكونات ضخمة.
* Backend معقد إلا إذا احتاج نموذج التواصل.

# 23. Contact Handling

## القرار المرحلي

نموذج التواصل يجب أن يرسل وصف المشكلة إلى قناة يمكن مراجعتها بسهولة.

الخيارات التنفيذية تحسم لاحقاً، لكن الـ UI يجب أن يدعم:

* Form submission.
* WhatsApp fallback.
* مرفقات اختيارية إذا أمكن.
* رسالة نجاح واضحة.
* رسالة خطأ واضحة.

## Success Message Direction

لا تقول:

تم استلام طلب مشروعك.

بل تقول:

تم استلام وصف المشكلة، وسنراجعه لفهم الوضع الحالي.

# 24. SEO Direction

## التركيز

SEO يجب أن يدعم السوق المحلي.

## كلمات عامة مناسبة

* أدوات تشغيلية للشركات العراقية.
* تطبيقات داخلية للشركات.
* تنظيم ملفات الشركات.
* أتمتة التقارير.
* تقليل العمل اليدوي.
* أنظمة عربية للشركات.
* أدوات مخصصة للشركات في العراق.

## كلمات يجب عدم تصدّرها كهوية

* شركة برمجيات.
* تطوير تطبيقات.
* تصميم مواقع.
* SaaS.
* ERP.
* CRM.
* حلول رقمية.

يمكن استخدامها بحذر فقط إذا احتاج SEO لاحقاً، لكن لا تكون الرسالة الأساسية.

# 25. Accessibility / Usability Rules

## يجب

* تباين جيد.
* حجم خط مريح.
* أزرار واضحة.
* حقول نموذج سهلة.
* رسائل خطأ مفهومة.

صممه لشخص يريد أن يفهم بسرعة:

هل هذه الجهة تفهم مشكلتي؟

# 26. Build Anti-Patterns

يجب رفض أي تنفيذ يجعل الموقع يبدو كـ:

* SaaS Landing Page.
* Software Company Site.
* Digital Agency Site.
* Freelancer Portfolio.
* Startup Website.
* Warehouse-only Solution.
* NAS FM technical sub-page.

## ممنوع تحديداً

* Dashboard Hero.
* Portfolio grid.
* Pricing cards.
* Feature matrix.
* Code visuals.
* App store style.
* Heavy gradients.
* Animated blobs.
* Stock team photos.
* Tech stack section.
* “Our Products”.
* “Our Packages”.
* “Request a Quote”.

# 27. Pre-Build Checklist

قبل بدء التنفيذ يجب التأكد من:

* النصوص النهائية معتمدة.
* CTA Architecture معتمدة.
* Navigation Architecture معتمدة.
* Visual Identity Finalization معتمدة.
* Visual Identity Update مع Operational Center معتمد.
* سعر Automation غير منشور.
* لا توجد خدمات جديدة.
* لا يوجد Hero Dashboard.
* Contact form لا يحتوي Budget.
* Case Studies ليست Portfolio.
* التصميم يبدأ من المشكلة.

---

# 28. Final Website UI / Build Principle

كل قرار UI أو Build يجب أن يخدم السؤال التالي:

هل يساعد الزائر على رؤية أن معلومات شركته المتفرقة يمكن أن تتحول إلى مركز رؤية تشغيلي أوضح؟

إذا نعم، فهو مناسب.

إذا جعل الموقع يبدو كمنصة، شركة برمجة، وكالة، Portfolio، أو صفحة طلب مشروع، فهو مرفوض.
